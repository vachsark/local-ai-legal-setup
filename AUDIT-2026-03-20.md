### 5. Citation Checker Bug — Fake Reporters Not Caught

**Issue**: `F.5th`, `F.6th`, `N.W.3d`, `S.E.3d` and other non-existent reporters were listed in `self.fake_reporters` but were NOT included in the reporter pattern used by `_find_case_citations()`. The fake reporter check only ran against citations that had already matched a known reporter. Result: fabricated citations using fake reporters silently passed through undetected.

**Fix** (`tools/citation-checker.py`): Changed `all_reporters = self.federal_reporters + self.state_reporters` to `all_reporters = self.federal_reporters + self.state_reporters + self.fake_reporters`. Fake reporters are now matched first, then immediately flagged.

**Verified**: `999 F.5th 100 (9th Cir. 2025)` now returns `[RED: Non-existent reporter: F.5th]`.

### 6. Output Validator — New Method on CitationChecker

Added `validate_output(output_json: str) -> dict` to `Tools` in `citation-checker.py`. ### 7. legal-check Shell Script — Improved think-stripping

**Issue**: `sed '/^<think>$/,/^<\/think>$/d'` only matched think tags on their own line, with no surrounding content. This would miss `<think>This is reasoning...</think>` in the same block.

**Fix**: Changed to `sed '/<think>/,/<\/think>/d'` which strips any line between (and including) `<think>` and `</think>` regardless of surrounding whitespace or inline content. With `/no_think` in all Qwen3.5 TEMPLATE blocks, this should never trigger in normal operation — it's a safety fallback.

# Audit Report — Cycle 2

**Date**: 2026-03-21
**Scope**: All Modelfiles (gap analysis vs. cycle 1), citation-checker.py (fake reporter coverage), legal-readability-scorer.py (document-type targeting)

---

## Findings & Changes

### 1. Modelfile.depo-prep — Missing TEMPLATE Block (Critical Bug)

**Issue**: `Modelfile.depo-prep` uses `FROM qwen3.5:9b` but had no TEMPLATE block and no `/no_think` suppression. The previous cycle fixed this for `contract-reviewer` and `clause-identifier` but missed `depo-prep`. Without think suppression, qwen3.5:9b spends the entire `num_predict 8192` budget on `<think>` chains and returns empty output.

**Fix**: Added the standard TEMPLATE block with `/no_think` in the user turn. Also added missing `top_p 0.9` and `repeat_penalty 1.1` parameters for consistency.

---

### 2. Modelfile.depo-summarizer — Missing num_predict and Sampling Parameters

**Issue**: `Modelfile.depo-summarizer` (gemma3:12b) had only `temperature` and `num_ctx`. Missing `num_predict`, `top_p`, and `repeat_penalty`. Without `num_predict`, Ollama uses the model default (varies by build — often 128 or 512, which truncates deposition summaries mid-section).

**Fix**: Added `num_predict 4096`, `top_p 0.9`, `repeat_penalty 1.05`. Used 1.05 (not 1.1) to match the prose-model convention — gemma3:12b is used for summarization, not structured JSON, so a lighter repeat penalty is appropriate.

---

### 3. Modelfile.court-rules — num_predict Too Low

**Issue**: `num_predict 2048` is insufficient for court-rules responses that include full sample certificate text, multi-step fix instructions, and circuit-specific variations. The standard certificate of compliance text alone is ~150 words. A complete rule explanation with fix steps regularly exceeds 2048 tokens.

**Fix**: Raised `num_predict 2048 → 4096`.

---

### 4. Modelfile.contract-reviewer — Added WRONG/CORRECT Contrast Example

**Issue**: The system prompt had correct output format instructions but no negative example. Research from vault autoresearch sessions shows models need negative contrast examples to avoid vague, non-actionable output. Without a "don't do this" anchor, models produce output like "Risk: HIGH — This section has risks." which gives attorneys nothing to act on.

**Fix**: Added a WRONG/CORRECT contrast block at the end of the system prompt, immediately before the closing `"""`. The WRONG example shows the vague pattern to avoid; the CORRECT example demonstrates the specificity required: quoting exact language, naming the mechanism of risk (uncapped + no liability cap interaction), stating missing protections, and providing a specific contractual fix.

---

### 5. citation-checker.py — Extended Fake Reporter Coverage

**Issue**: The `fake_reporters` list was missing the most common LLM hallucination patterns for state reporters: non-existent next-series reporters. LLMs routinely fabricate `Cal.App.5th` (series tops at 4th), `N.Y.S.4d` or `N.Y.S.4th` (tops at 3d), `P.4th` (Pacific tops at 3d), `S.W.4th` (Southwestern tops at 3d), `So.4th` (Southern tops at 3d), `A.4th` (Atlantic tops at 3d), and `Cal.Rptr.4th` (tops at 3d). These passed silently through the checker.

**Fix**: Added 9 new fake reporter patterns with inline comments explaining the current highest series for each. Added comment block explaining the "next series" hallucination pattern for future maintainers.

New patterns added: `S.W.4th`, `P.4th`, `A.4th`, `So.4th`, `N.Y.S.4d`, `N.Y.S.4th`, `Cal.App.5th`, `Cal.App.6th`, `Cal.Rptr.4th`.

---

### 6. legal-readability-scorer.py — Document-Type Target Bands

**Issue**: The scorer returned grade-level interpretation bands but had no way to tell the user whether their grade level was appropriate for their _specific document type_. Grade 14 is "on target" for a brief but "too dense" for a consumer disclosure. The previous output gave the same generic interpretation regardless.

**Fix**: Added optional `document_type` parameter with four options: `"brief"`, `"contract"`, `"client_letter"`, `"consumer"`, `"auto"` (default). When specified (or auto-detected from grade level), the output now includes:

- `document_type_target`: ON TARGET / BELOW TARGET / ABOVE TARGET with grade range
- `document_type_guidance`: Document-type-specific actionable note

Target bands are grounded in ABA plain-language research and CFPB consumer disclosure standards (grade 8 or below for consumer-facing disclosures).

Auto-detection logic: grade ≥16 → brief, ≥12 → contract, ≥9 → client_letter, else → consumer.

Backward-compatible: default is `"auto"`, existing callers see no breaking change in the required fields.

---

## Remaining Gaps (Not Fixed This Cycle)

- **Modelfile.legal-grammar-ft**: The fine-tuned model system prompt is intentionally compressed (it's a GGUF with baked-in weights). Edge-case handling (empty input, non-English text) was not added — the grammar checker is expected to receive non-empty English legal text as a precondition. Adding guards would bloat a prompt that's already optimized for a fine-tuned model.
- **legal-grammar-checker.py**: The tool wraps LanguageTool and correctly passes all 47+ rule categories through. The fine-tuned model integration is a separate pathway (Modelfile.legal-grammar-ft) — the LanguageTool-based checker is not redundant. No changes needed.
- **Non-US citations** (UK EWCA/EWHC, Canada SCR/FCR): Not added — the project README scopes this tool for US legal work. Adding UK/Canadian patterns without testing would increase false positives. Recommend as a future issue if international work is in scope.
- **Parallel citations**: The `_find_case_citations` pattern captures the first citation in a parallel pair but may miss the second (e.g., `550 U.S. 544, 127 S. Ct. 1955`). The `v.`-pattern check will catch the case name. Acceptable coverage given "high recall" design philosophy.

## Context

## Changes Applied

### 1. validate_output() — batch-scanner.py

Added `validate_output(report: str) -> dict` to the `Tools` class.

**Pattern source**: vault quality scoring system (citation-checker.py already had this from cycle 1; batch-scanner was missing it).

**What it checks**:

- Report is non-empty and starts with a recognized header
- At least one `### DocName` section present (catches zero-document directories)
- `## Summary Table` and `## Detailed Findings` sections present in risk reports
- Truncation artifacts from Ollama `num_predict` cuts (`[AI analysis timed out`, etc.)

**Use case**: Pipeline callers consuming batch-scanner output programmatically (e.g., a heartbeat task that scans a contracts folder and summarizes HIGH-risk findings). Without validation, a truncated or empty report silently propagates downstream.

---

### 2. validate_output() — court-rules-checker.py

Added `validate_output(report: str) -> dict` to the `Tools` class.

**Pattern source**: same vault quality scoring pattern.

**What it checks**:

- Header present (`## Filing Compliance Check — <jurisdiction>`)
- Error string detection (catches `Error: jurisdiction not found` being returned as a "report")
- Overall verdict line present (`**Overall: PASS/FAIL/REVIEW**`)
- At least one ✓/✗/⚠ check result line (catches truncation before checks ran)
- Word count line present (structural completeness signal)

---

### 3. Document boundary markers — batch-scanner.py

Added `<!-- [DOCUMENT: name] -->` / `<!-- [END: name] -->` HTML comment delimiters around each document's section in `format_risk_report` and `format_privilege_report`.

**Pattern source**: document-qa.py's `[DOCUMENT: filename | SOURCE: citation]` / `[END: citation]` boundary markers (already in the codebase for single-document Q&A).

**Why**: batch-scanner generates multi-document reports that can exceed 30k chars for 20+ contract directories. LLMs consuming these reports to answer "which contracts have uncapped indemnification?" need unambiguous per-document delimiters. HTML comments don't affect markdown rendering but are parse-visible.

## vault-search OSS (github.com/vachsark/vault-search)
